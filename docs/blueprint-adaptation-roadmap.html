1 · สรุปผู้บริหาร

Blueprint ที่ ~/Downloads/blueprint/ คือเอกสารของ SCG Togo / CPAC ToGo — marketplace ขายคอนกรีตผสมเสร็จ (RMC) ที่ขึ้น production แล้ว ลูกค้าสั่งผ่าน LINE (PHP LIFF app) ระบบหาโรงงานที่ใกล้และผ่านเงื่อนไขที่สุด คิดราคาจาก pricelist ตามจังหวัด/วันที่/ระยะทาง/ประเภทรถ รับเงินผ่าน 2c2p แจ้งโรงงานผ่าน LINE Messaging ประสานการส่งพร้อมรูปยืนยัน แล้ว จ่าย payout ให้โรงงานหรือบริษัทแม่ CPAC หลังหักค่าธรรมเนียม 2% + VAT เป็นระบบ 4 ส่วน หลังบ้าน MySQL 65 ตาราง

ทำไม map กับ Thai Agri Market ได้: ทั้งคู่เป็น managed marketplace ที่ platform ไม่ถือ stock — operator คัดสวน (= company/plant) เปิด lot ตาม harvest window (= product) รับเงิน แล้วจ่ายที่เหลือให้สวนหลังหักค่าธรรมเนียม แกนเงิน (order → escrow/hold → payout → refund), running-number กันชนกัน, order lifecycle, reschedule/adjust, รูปส่ง + review, RBAC admin, LINE LIFF + Messaging นำมาใช้ซ้ำตรงๆ ความต่างเชิงโครงสร้างคือ logistics: SCG Togo จับคู่โรงงานตามระยะทาง (geofence/30 กม. + Longdo) ส่วนเราเป็นผลไม้ต่างจังหวัดส่งโดย logistics partner เข้ากรุงเทพ — distance band, แบ่งคิวรถ, slump/strength/structure ตัดทิ้งหมด

คำแนะนำหลัก: ใช้ blueprint เป็น เอกสารอ้างอิงเชิงพฤติกรรมและ data model ไม่ใช่ผู้บริจาค code สร้างใหม่บน Next.js + Prisma + Postgres เก็บ pattern เงิน/concurrency/notification/RBAC ตัดทุกอย่างที่เป็น physics คอนกรีต ลำดับเป็น vertical slice บางก่อน (Phase 1) แล้วซ้อน fulfillment, settlement, surface สร้าง customer surface ทีละตัว (LIFF → web → Expo)

⚠ ข้อควรระวัง build-gate (เจ้าของตัดสิน — อย่าถือว่า override)
README.md + vault operator-master-plan.md (Gate 4) ระบุว่า software build ถูก block จนกว่า: (1) ทำ paid drop เทียบกันได้ 3 ครั้ง, (2) เห็น paid repeat behavior, (3) contribution ระดับ lane เป็นบวกหลังหัก variable cost, (4) cash reconciliation ปิดได้, (5) วัด manual pain ซ้ำๆ ได้ — บวก Gate 0 (counsel, PSP, accountant, logistics SLA, ซ้อม pack/QC) roadmap นี้คือแผนพร้อม build แต่ activate ก็ต่อเมื่อเจ้าของเคลียร์ gate เท่านั้น

9.4 · เกณฑ์ตรวจรับ

  1. LINE user ไม่ verify สร้าง order ไม่ได้ (gate ฝั่ง server)
  2. orderNo = <prefix>+YYMMDD+3digit unique ภายใต้สร้างพร้อมกัน (hammer test, bug #2)
  3. order ไม่จ่าย expire หลัง 1h → EXPIRED
  4. callback HMAC ผิด → reject, ไม่เขียน DB
  5. callback ถูก → Order=PAID+Payment=COMPLETED/HELD+PaymentCallbackLog แบบ atomic
  6. admin login เห็น order paid, RBAC-scoped
  7. push LINE ยิงตอน payment สำเร็จ ผ่าน internal /push/*

11.2 · ความเสี่ยง domain-mismatch + build ใหม่

  • Cron serverless — Next.js/Vercel ไม่มี process รันยาว ใช้ Vercel Cron/scheduled function; เก็บ dedup cron_logs (idempotency) · sweep expiry ต้องใช้ตั้งแต่ P1
  • Money atomicity — README note 1 สำคัญสุด: ครอบทุก money state ด้วย prisma.$transaction · drift = สูญเงินจริง
  • HMAC callback seam — PSP/LINE callback เป็น surface เดียวไม่ auth JWT · signature check หาย/หลวม = ช่อง exploit · verify ก่อน read/write
  • Multi-lot cart — ผลไม้ต้อง OrderItem หลาย lot · refund/adjust เป็นราย-item · ออกแบบที่ grain item แต่แรก
  • QC/claim/reconciliation = net-new — blueprint ไม่มีคู่ อย่า force-fit · human-gate · หัวใจ operation ผลไม้ · เผื่อเวลาออกแบบ (P2/P6/P7)
  • ไม่มี CPAC/franchise split, แบ่งคิวรถ, distance band — ต้านการ copy; เพิ่มความซับซ้อนโดยไม่มีค่าผลไม้
